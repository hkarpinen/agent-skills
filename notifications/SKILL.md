## Notification Domain Model

A notification is a record that a relevant event occurred for a specific user. It is not the event itself — it is a **user-facing projection** of a domain event.

### Aggregate: Notification

```
Notification (aggregate root)
    Id: NotificationId
    RecipientId: UserId         // who receives it
    Type: NotificationType      // reply, mention, upvote, system
    Title: string               // "Jane replied to your post"
    Body: string?               // optional preview text
    ResourceType: string        // "post", "thread"
    ResourceId: Guid            // the entity the notification is about
    IsRead: bool
    CreatedAt: datetime
    ReadAt: datetime?

    MarkAsRead()
        if IsRead: return
        IsRead = true
        ReadAt = utcNow()
```

### Notification Types

| Type | Trigger Event | Example Title |
|---|---|---|
| `reply` | `PostCreated` (where thread author ≠ post author) | "Jane replied to your thread" |
| `mention` | `PostCreated` (body contains `@username`) | "Jane mentioned you in a post" |
| `upvote` | `VoteCast` (on user's content) | "Your post received an upvote" |
| `thread_locked` | `ThreadLocked` (notify thread author) | "Your thread was locked by a moderator" |
| `system` | Admin broadcast | "Welcome to the forum!" |

## Storage

### Database Table

SQL examples use PostgreSQL syntax (`timestamptz`, partial indexes). Adapt types for other databases.

```sql
CREATE TABLE notifications.notifications (
    id            uuid         NOT NULL,
    recipient_id  uuid         NOT NULL,
    type          text         NOT NULL,
    title         text         NOT NULL,
    body          text,
    resource_type text,
    resource_id   uuid,
    is_read       boolean      NOT NULL DEFAULT false,
    created_at    timestamptz  NOT NULL DEFAULT now(),
    read_at       timestamptz,
    CONSTRAINT pk_notifications PRIMARY KEY (id)
);

CREATE INDEX ix_notifications_recipient_unread
    ON notifications.notifications (recipient_id, created_at DESC)
    WHERE is_read = false;

CREATE INDEX ix_notifications_recipient_all
    ON notifications.notifications (recipient_id, created_at DESC);
```

Rules:
- Index on `(recipient_id, created_at DESC)` for the common query: "show me my latest notifications."
- Partial index on `is_read = false` for the badge count query: "how many unread notifications do I have?"
- Soft-delete or hard-delete old notifications (e.g., > 90 days) to prevent unbounded table growth.
- Notifications live in their own schema (`notifications`) if using schema-per-context, or their own database if using database-per-context.

## Notification Generation

Notifications are generated by **event consumers** — they subscribe to domain events and create notification records for affected users.

```
Domain Event: PostCreated { postId, threadId, authorId }

NotificationConsumer:
    1. Load thread to get thread.authorId
    2. If authorId ≠ post.authorId:
       Create Notification(recipientId=thread.authorId, type=reply, ...)
    3. Parse post body for @mentions
    4. For each mentioned user:
       Create Notification(recipientId=mentionedUserId, type=mention, ...)
```

Rules:
- Never generate notifications synchronously in the request path. Use an event consumer (async, via message broker or in-process event dispatcher).
- A user should never receive a notification for their own action (posting, voting).
- Deduplicate: if the same event would create duplicate notifications (e.g., user is both the thread author and @mentioned), create only one.
- Notification consumers are idempotent — processing the same event twice does not create duplicate notifications. Use a deduplication key: `(recipientId, type, resourceId)`.

## Aggregation and Batching

Avoid notification fatigue by aggregating related events.

| Strategy | Example |
|---|---|
| **Aggregate by resource** | "5 people replied to your thread" instead of 5 separate notifications |
| **Aggregate by type** | "You received 10 upvotes today" instead of 10 notifications |
| **Throttle per user** | Max 1 email notification per 15 minutes; batch pending notifications |

Rules:
- For in-app notifications, show individual notifications (users expect real-time granularity).
- For email notifications, aggregate into a digest. Do not send an email per event.
- Store individual notification records even when displaying aggregated summaries — the user can expand to see details.

## Delivery Channels

### In-App (Primary)

- Query `/api/notifications?unread=true` for the notification dropdown.
- Query `/api/notifications/count` for the badge number.
- Use cursor-based pagination for the full notification list.
- Mark as read: `PUT /api/notifications/{id}/read` or `PUT /api/notifications/read-all`.

### Email (Secondary)

Rules:
- Email is opt-in per notification type. Store user preferences:
  ```sql
  CREATE TABLE notifications.preferences (
      user_id             uuid    NOT NULL,
      notification_type   text    NOT NULL,
      email_enabled       boolean NOT NULL DEFAULT true,
      CONSTRAINT pk_preferences PRIMARY KEY (user_id, notification_type)
  );
  ```
- Never send email synchronously. Enqueue email jobs to a background processor.
- Use a transactional email service (SendGrid, Postmark, AWS SES) — do not run your own SMTP server.
- Include an unsubscribe link in every email (legally required in many jurisdictions).
- Throttle email delivery: max 1 digest email per configurable interval (e.g., 15 minutes).

### Email Infrastructure

Use a transactional email service — do not run your own SMTP server.

| Provider | Strength |
|---|---|
| **Postmark** | Best deliverability, simple API, focused on transactional email |
| **SendGrid** | High volume, marketing + transactional in one platform |
| **AWS SES** | Cheapest at scale, requires more setup |
| **Resend** | Developer-friendly API, React Email templates |

Architecture:

```
Domain Event → Notification Consumer → Notification Record (DB)
                                          ↓
                              Email Job (background queue)
                                          ↓
                              Email Service (SendGrid/Postmark SDK)
```

Rules:
- Send email via a **background job**, never in the request path. Email APIs have variable latency (100ms–2s).
- Use the provider's SDK, not raw SMTP. SDKs handle retries, connection pooling, and error parsing.
- Template emails server-side. Pass variables to the provider's template engine or render HTML with a library (e.g., React Email, Mjml) before sending.
- Store a record of sent emails (recipient, template, timestamp) for debugging and audit.
- Handle bounces and complaints via the provider's webhook — update the user's email preference to stop sending.
- Rate-limit email sending per user (max 1 digest per interval) even if multiple events trigger notifications.

### Push (Future)

Web push notifications via the Push API. Out of scope for MVP — add when in-app + email are established.

## API Endpoints

| Verb | Path | Description |
|---|---|---|
| GET | `/api/notifications` | List notifications (cursor-paginated, filterable by read/unread) |
| GET | `/api/notifications/count` | Unread count (for badge) |
| PUT | `/api/notifications/{id}/read` | Mark one as read |
| PUT | `/api/notifications/read-all` | Mark all as read |
| GET | `/api/notifications/preferences` | Get notification preferences |
| PUT | `/api/notifications/preferences` | Update notification preferences |

## Where Notifications Live Architecturally

Notifications are a **cross-cutting concern**, not a bounded context with deep domain logic. Two valid placements:

| Approach | When |
|---|---|
| **Within the Forum bounded context** | Notifications are only for forum events; simple system |
| **Separate Notifications context** | Multiple bounded contexts generate notifications (Forum, Billing, Identity); notification preferences and delivery are complex enough to warrant isolation |

For a system with Identity + Forum, start with notifications inside the Forum context. Extract to a separate context when a third context needs to send notifications.
