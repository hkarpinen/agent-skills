# db-postgres

> PostgreSQL schema design conventions — how to translate domain models into PostgreSQL idioms.

**When to use this skill**: designing or reviewing a PostgreSQL database schema, choosing
data types, defining indexes, constraints, or schema separation. Stack and ORM agnostic —
applies regardless of whether the application layer is .NET, Java, Python, or any other platform.

**Composes with**: `righting-software` (produces the domain model this skill translates),
and a bridge skill (e.g. `dotnet-efcore-postgres`) that maps these conventions to a specific ORM.

**Scope**: this skill translates domain model concepts into PostgreSQL. It does not design
the domain model — that is `righting-software`'s job. Given an aggregate, entity, or value
object, this skill answers: how should it be represented in PostgreSQL?

---

## Core Philosophy

- The schema serves the domain model — not the other way around.
- Schema boundaries should reflect the volatility boundaries in the domain, not be driven
  by ORM convenience or database normalization alone.
- Use PostgreSQL-native features where they encode domain rules better than generic SQL.

---

## Naming Conventions

All identifiers use `snake_case`. This is the PostgreSQL convention and avoids the need
to quote identifiers in queries.

| Concept | Convention | Example |
|---|---|---|
| Schema | `snake_case`, domain noun | `orders`, `identity`, `billing` |
| Table | `snake_case`, plural noun | `orders`, `order_lines`, `customers` |
| Column | `snake_case`, singular | `customer_id`, `created_at`, `total_amount` |
| Primary key column | always `id` | `id` |
| Foreign key column | `{referenced_table_singular}_id` | `order_id`, `customer_id` |
| Index | `ix_{table}_{columns}` | `ix_orders_customer_id` |
| Unique constraint | `uq_{table}_{columns}` | `uq_customers_email` |
| Check constraint | `ck_{table}_{rule}` | `ck_order_lines_quantity_positive` |
| Primary key constraint | `pk_{table}` | `pk_orders` |
| Foreign key constraint | `fk_{table}_{referenced}` | `fk_order_lines_orders` |

Never use reserved words as identifiers. Avoid `type`, `name`, `value`, `user`, `order`
as bare column names — prefix with the entity: `order_status`, `customer_type`.

## Schema Separation

Separate schemas by domain concern, mirroring the volatility boundaries in the domain model.
Never put everything in `public`.

```sql
CREATE SCHEMA IF NOT EXISTS orders;
CREATE SCHEMA IF NOT EXISTS customers;
CREATE SCHEMA IF NOT EXISTS billing;
CREATE SCHEMA IF NOT EXISTS identity;   -- auth / user management
CREATE SCHEMA IF NOT EXISTS audit;      -- domain events / audit trail
```

Rules:
- One schema per bounded domain concern or aggregate cluster.
- Infrastructure concerns (identity, audit) get their own schemas, never mixed with domain tables.
- Cross-schema foreign keys are a warning sign of a boundary problem in the domain model.
  Prefer referencing by ID and resolving joins in the application layer.
- `public` is reserved for shared extensions (`pgcrypto`, `uuid-ossp`, `pg_trgm`).

## Primary Keys

All tables use `uuid` as the primary key. UUIDs are generated by the application layer —
never by the database as a column default.

```sql
CREATE TABLE orders.orders (
    id  uuid  NOT NULL,
    -- ...
    CONSTRAINT pk_orders PRIMARY KEY (id)
);
```

Rationale: the aggregate root in the domain model owns its identity. Delegating ID
generation to the database makes identity a persistence concern rather than a domain concern,
and causes problems when records are created across multiple application instances before
being persisted.

## Timestamps

All timestamp columns use `timestamptz` (timestamp with time zone). Never use `timestamp`
without time zone — it silently discards timezone information.

```sql
created_at  timestamptz  NOT NULL  DEFAULT now(),
updated_at  timestamptz  NOT NULL  DEFAULT now(),
deleted_at  timestamptz  NULL                      -- nullable for soft delete
```

Rules:
- `timestamptz` stores UTC internally and converts on retrieval — always the correct behavior.
- `created_at` and `updated_at` are present on every table.
- `updated_at` is maintained by the application layer, not a database trigger.
- Use `deleted_at` for soft deletes — not a boolean `is_deleted` column (see Soft Deletes).

## Data Types

### Standard Type Choices

| Domain concept | PostgreSQL type | Notes |
|---|---|---|
| Aggregate identity / PK | `uuid` | Always generated by the application |
| Bounded text | `varchar(n)` | When max length is a domain rule |
| Unbounded text | `text` | Prefer over `varchar` without a meaningful length |
| Monetary amount | `numeric(19,4)` | Never `float` or `double precision` for money |
| Currency code | `char(3)` | ISO 4217 — `USD`, `GBP`, `EUR` |
| Boolean | `boolean` | Not `smallint` or `char(1)` |
| Whole number | `integer` | Use `bigint` for counters that may exceed 2 billion |
| Timestamp | `timestamptz` | Always — see Timestamps |
| Date only | `date` | Only when time is genuinely not a domain concern |

### PostgreSQL-Specific Types

#### jsonb
Use `jsonb` for structured data where the shape is volatile or genuinely schema-less.
Do not use it to avoid modelling — if the data has a stable shape, give it columns.

```sql
-- Domain events: payload shape varies by event type — jsonb is appropriate
CREATE TABLE audit.domain_events (
    id           uuid         NOT NULL,
    event_type   text         NOT NULL,
    payload      jsonb        NOT NULL,
    occurred_at  timestamptz  NOT NULL,
    CONSTRAINT pk_domain_events PRIMARY KEY (id)
);

-- GIN index on a frequently queried jsonb field
CREATE INDEX ix_domain_events_order_id
    ON audit.domain_events ((payload->>'order_id'))
    WHERE event_type = 'OrderConfirmed';
```

#### Enums
Use PostgreSQL enums for domain status values that are known at design time and change
rarely. If the value set changes frequently, use `text` with a check constraint instead
— altering an enum type requires a schema migration.

```sql
CREATE TYPE orders.order_status AS ENUM (
    'draft',
    'pending',
    'confirmed',
    'shipped',
    'cancelled'
);

CREATE TABLE orders.orders (
    id      uuid                 NOT NULL,
    status  orders.order_status  NOT NULL  DEFAULT 'draft',
    CONSTRAINT pk_orders PRIMARY KEY (id)
);
```

#### Arrays
Use arrays for small, bounded, homogeneous sets that are always read and written with
the parent row. Do not use arrays as a substitute for a related table when elements
need their own identity, metadata, or are queried individually.

```sql
-- Tags: simple string set, always accessed with the product
tags  text[]  NOT NULL  DEFAULT '{}',

-- GIN index for array membership queries
CREATE INDEX ix_products_tags ON catalog.products USING gin(tags);
```

#### Ranges
Use range types (`daterange`, `tstzrange`, `numrange`) for interval data where overlap
queries or exclusion constraints are needed.

```sql
-- Price periods: no overlapping periods for the same product
CREATE TABLE billing.price_periods (
    id            uuid          NOT NULL,
    product_id    uuid          NOT NULL,
    price         numeric(19,4) NOT NULL,
    valid_during  daterange     NOT NULL,
    CONSTRAINT pk_price_periods PRIMARY KEY (id),
    CONSTRAINT uq_price_periods_no_overlap
        EXCLUDE USING gist (product_id WITH =, valid_during WITH &&)
);
```

## Constraints

Define all constraints explicitly with named constraints. Never rely on ORM-generated
constraint names — they are unreadable in error messages and migration output.

```sql
CREATE TABLE orders.order_lines (
    id          uuid          NOT NULL,
    order_id    uuid          NOT NULL,
    product_id  uuid          NOT NULL,
    quantity    integer       NOT NULL,
    unit_price  numeric(19,4) NOT NULL,
    created_at  timestamptz   NOT NULL  DEFAULT now(),
    updated_at  timestamptz   NOT NULL  DEFAULT now(),

    CONSTRAINT pk_order_lines             PRIMARY KEY (id),
    CONSTRAINT fk_order_lines_orders      FOREIGN KEY (order_id)
                                              REFERENCES orders.orders (id)
                                              ON DELETE CASCADE,
    CONSTRAINT ck_order_lines_quantity    CHECK (quantity > 0),
    CONSTRAINT ck_order_lines_unit_price  CHECK (unit_price >= 0)
);
```

Foreign key `ON DELETE` actions — choose deliberately:
- `CASCADE` — child rows deleted with parent. Use for owned entities (order lines owned by order).
- `RESTRICT` — prevents parent deletion if children exist. Use to enforce referential integrity.
- `SET NULL` — orphans the child. Use only when orphaning is a valid domain state.

## Indexing Strategy

### Always Index
- Primary keys are indexed automatically.
- Every foreign key column needs an explicit index — PostgreSQL does not create these automatically.

```sql
CREATE INDEX ix_order_lines_order_id   ON orders.order_lines (order_id);
CREATE INDEX ix_order_lines_product_id ON orders.order_lines (product_id);
```

### Add Indexes For
- Columns in frequent `WHERE` clauses
- Columns in `ORDER BY` on large tables
- Columns in `JOIN` conditions beyond FKs

Do not add indexes speculatively. Index on measured query patterns.

### Index Types

| Use case | Type |
|---|---|
| Equality and range queries (default) | B-tree |
| `jsonb` field queries | GIN |
| Array membership (`@>`, `&&`) | GIN |
| Full-text search | GIN on `tsvector` |
| Range overlap, geometric | GiST |

### Partial and Covering Indexes

```sql
-- Partial index: only active orders (avoids indexing historical data)
CREATE INDEX ix_orders_customer_active
    ON orders.orders (customer_id)
    WHERE status NOT IN ('shipped', 'cancelled');

-- Covering index: include columns to satisfy query from index alone
CREATE INDEX ix_orders_customer_covering
    ON orders.orders (customer_id)
    INCLUDE (status, created_at);
```

## Normalization

- Normalize to 3NF by default. Denormalize only when there is a measured performance
  justification — document the reason in the migration.
- Value objects from the domain model map to columns on the owning entity's table, not
  to separate tables. `Money(amount, currency)` becomes `total_amount numeric(19,4)` and
  `total_currency char(3)` on the parent table.
- Many-to-many relationships always use an explicit junction table with its own `uuid` PK,
  `created_at`, and any relationship-specific attributes.
- Never store comma-separated values in a text column. Use an array, junction table, or
  related table.

## Soft Deletes

```sql
-- Partial index excludes deleted rows from normal queries
CREATE INDEX ix_orders_active
    ON orders.orders (customer_id, created_at)
    WHERE deleted_at IS NULL;

-- View for application queries
CREATE VIEW orders.active_orders AS
    SELECT * FROM orders.orders
    WHERE deleted_at IS NULL;
```

---

## Partitioning

Partition large tables by time range when the table will exceed tens of millions of rows
and queries are almost always scoped to a recent window. Use range partitioning on
`created_at`. Apply only after measuring — partitioning adds operational complexity.

## Security and Permissions

The application runtime user should not own schema objects.

```sql
-- Migration/CI user owns the schema
GRANT ALL ON SCHEMA orders TO migration_user;

-- Application runtime user: DML only, no DDL
GRANT USAGE ON SCHEMA orders TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE
    ON ALL TABLES IN SCHEMA orders TO app_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA orders
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;
```
